Lenguaje de Marcas

Unidad didáctica — 1º DAW · Estructura, semántica y validación

¿Qué es un Lenguaje de Marcas?

Un lenguaje de marcas utiliza etiquetas (marcas) para describir la estructura, el significado y, a veces, la presentación de la información en un documento. No ejecuta operaciones como un lenguaje de programación: su propósito es organizar datos y contenido de forma que persona y máquina entiendan la jerarquía, el contexto y la relación entre partes.

Idea clave: con marcado, un mismo contenido se vuelve estructurado, semántico y procesable. La capa visual se delega a otras tecnologías (por ejemplo, CSS en el caso de HTML).

Tipos y usos habituales

  • HTML: describe la estructura de páginas web (títulos, párrafos, enlaces, formularios...).
  • XML: describe datos intercambiables y autodescriptivos entre sistemas.
  • Markdown: formato ligero de texto que se convierte a HTML, muy usado en documentación.

Comparativa rápida

LenguajeObjetivo principalVentajasConsideraciones
HTML Estructura y contenido web Estándar de la web; semántica rica; accesible No es para datos arbitrarios; presentación con CSS
XML Datos estructurados e intercambio Extensible; validable; autodescriptivo Verborrea; requiere definir reglas (DTD/XSD)
Markdown Redacción ligera y legible Curva de entrada corta; ideal para documentación Variantes (GFM, CommonMark); limitado sin extensiones

Consejo: separa responsabilidades: el marcado define significado/estructura; el estilo (CSS) define apariencia; la lógica (JS) define interactividad.

XML

XML (eXtensible Markup Language) es un lenguaje de marcas diseñado para describir e intercambiar datos. Su fortaleza es la extensibilidad: tú defines las etiquetas adecuadas a tu dominio (catálogo, pacientes, pedidos…). XML no decide cómo se presenta; describe qué es cada dato.

Bien formado vs válido

  • Bien formado: cumple las reglas sintácticas de XML (cierre de etiquetas, anidación correcta, un único elemento raíz, atributos entre comillas, distinción de mayúsculas/minúsculas).
  • Válido: además de estar bien formado, cumple un conjunto de reglas declaradas en un DTD o un XML Schema (XSD).

Estructura básica

<?xml version="1.0" encoding="UTF-8"?>
<catalogo>
  <producto id="P-001">
    <nombre>Teclado mecánico</nombre>
    <precio moneda="EUR">89.90</precio>
  </producto>
</catalogo>

Prolog y codificación

  • <?xml version="1.0" encoding="UTF-8"?>: declara versión y charset. UTF-8 es la opción recomendada.
  • standalone="yes|no": indica si el documento depende de declaraciones externas (DTD).

Entidades y secciones CDATA

  • Escapa caracteres especiales: &lt;, &gt;, &amp;, &quot;, &apos;.
  • Para incluir texto sin parsear (p. ej., trozos de código con <): usa <![CDATA[ ... ]]>.

Nombres y espacios de nombres (namespaces)

Para evitar colisiones, XML permite definir namespaces con xmlns. Se antepone un prefijo a los nombres calificados: 

<rss xmlns:dc="http://purl.org/dc/elements/1.1/">
  <item>
    <title>Noticia</title>
    <dc:creator>Redacción</dc:creator>
  </item>
</rss>

Errores comunes: múltiples elementos raíz, etiquetas sin cerrar, mezcla de mayúsculas/minúsculas, atributos sin comillas, uso incorrecto de namespaces.

¿Atributos o elementos?

  • Usa elementos para contenido estructurado o repetible: <direccion>, <linea>.
  • Usa atributos para metadatos del elemento: identificadores, unidades, idioma, etc.

Ejemplos prácticos

Ejemplo 1 · Catálogo con atributos y CDATA

Demostración de atributos para metadatos y uso de CDATA para texto “sin interpretar”.

<catalogo fecha="2025-09-18" version="1.2">
  <producto id="P-001" activo="true">
    <nombre>Teclado mecánico</nombre>
    <descripcion><![CDATA[
      Teclado con switches <clicky> y retroiluminación.
      Compatibilidad: Windows, Linux, macOS.
    ]]></descripcion>
    <precio moneda="EUR">89.90</precio>
    <tags><tag>periférico</tag><tag>teclados</tag></tags>
  </producto>
</catalogo>

Ejemplo 2 · Contenido mixto (texto + elementos)

Útil cuando el texto fluye alrededor de marcas (citas, énfasis, referencias).

<parrafo>Esta es una <enfasis nivel="fuerte">nota importante</enfasis> 
que conviene leer con <ref id="R-12">atención</ref>.</parrafo>

Ejemplo 3 · Namespaces y enlaces

<doc xmlns:xlink="http://www.w3.org/1999/xlink">
  <recurso xlink:href="https://ejemplo.com/spec" xlink:title="Especificación"/>
</doc>

Validación de XML

Validar un XML significa comprobar que su estructura y valores cumplen unas reglas. Las dos vías más comunes son DTD (histórica y ligera) y XSD (potente y tipada).

DTD (Document Type Definition)

Define qué elementos puede tener un documento, su orden y cardinalidad, y los atributos permitidos.

  • Modelos de contenido: (a,b) secuencia; (a|b) elección.
  • Cuantificadores: * (0..n), + (1..n), ? (0..1).
  • Tipos de atributo: CDATA, ID, IDREF, ENUMERATION…

DTD interno · Biblioteca

<!DOCTYPE biblioteca [
  <!ELEMENT biblioteca (libro+)>
  <!ELEMENT libro (titulo, autor, genero, anio)>
  <!ATTLIST libro isbn ID #REQUIRED>
  <!ELEMENT titulo (#PCDATA)>
  <!ELEMENT autor (#PCDATA)>
  <!ELEMENT genero (#PCDATA)>
  <!ELEMENT anio (#PCDATA)>
]>
<biblioteca>
  <libro isbn="ISBN-001">
    <titulo>XML práctico</titulo>
    <autor>A. López</autor>
    <genero>Técnico</genero>
    <anio>2024</anio>
  </libro>
</biblioteca>

Ten en cuenta: ID debe ser único en todo el documento; IDREF debe referenciar a un ID existente.

XML Schema (XSD)

XSD añade tipos de datos, restricciones (facets) y construcción de tipos complejos. Admite namespaces, claves/relaciones (xs:key, xs:keyref) y reutilización de tipos.

XSD con tipos y restricciones

<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema"
           elementFormDefault="qualified">

  <!-- Tipo simple restringido -->
  <xs:simpleType name="isbnType">
    <xs:restriction base="xs:string">
      <xs:pattern value="ISBN-\d{3}"/>
    </xs:restriction>
  </xs:simpleType>

  <!-- Tipo complejo libro -->
  <xs:complexType name="LibroType">
    <xs:sequence>
      <xs:element name="titulo" type="xs:string"/>
      <xs:element name="autor" type="xs:string"/>
      <xs:element name="genero" type="xs:string"/>
      <xs:element name="anio" type="xs:gYear"/>
    </xs:sequence>
    <xs:attribute name="isbn" type="isbnType" use="required"/>
  </xs:complexType>

  <!-- Elemento raíz -->
  <xs:element name="biblioteca">
    <xs:complexType>
      <xs:sequence>
        <xs:element name="libro" type="LibroType" maxOccurs="unbounded"/>
      </xs:sequence>
    </xs:complexType>
    <!-- Clave única por ISBN -->
    <xs:key name="k_isbn">
      <xs:selector xpath="libro"/>
      <xs:field xpath="@isbn"/>
    </xs:key>
  </xs:element>
</xs:schema>

Uso en documentos: referencia tu XSD con xsi:noNamespaceSchemaLocation="biblioteca.xsd" o xsi:schemaLocation="URI-NS ruta.xsd" cuando trabajes con namespaces.

Markdown

Markdown es un lenguaje de marcado ligero pensado para escribir con rapidez y convertir a HTML (o PDF/Docx mediante herramientas como Pandoc). Su sintaxis es mínima y legible en texto plano, lo que lo hace ideal para documentación técnica, apuntes y README en repositorios.

Sintaxis esencial

# Título de nivel 1
## Título de nivel 2

Texto en **negrita** y *cursiva*. Código en `inline`.

- Lista no ordenada
1. Lista ordenada

[Enlace](https://ejemplo.com)
![Alt descriptivo](img/diagrama.png)

Funciones útiles (GFM)

  • Tablas: 
    | Campo | Tipo |
|------:|:----:|
| id    | int  |
  • Listas de tareas: 
    - [x] Definir esquema
- [ ] Escribir XSD
- [ ] Validar ejemplos
  • Bloques de código: 
    ```xml
<persona><nombre>Ana</nombre></persona>
```

Ten en cuenta: existen variantes de Markdown (CommonMark, GFM). Algunas funciones (tablas, tareas, notas al pie) dependen del motor que procese el Markdown.

Ejercicios propuestos

XML · Contactos con DTD

Enunciado: crea un XML <contactos> con elementos <contacto> (nombre, teléfono, email). Define un DTD que exija al menos un contacto, fuerce el patrón del email y marque el teléfono como obligatorio.

Pistas: usa ID para un identificador único; en DTD, los patrones complejos se suelen resolver por convenio (o se valida con XSD).

  • XML bien formado y válido con DTD.
  • Al menos 3 contactos.
  • Entrega: contactos.xml y contactos.dtd.

XML/XSD · Biblioteca tipada

Enunciado: modela una <biblioteca> con <libro> (isbn, título, autor, género, año). Crea un XSD que exija formato de ISBN (ISBN-123), anio como gYear y género como enumeración fija.

Pistas: xs:pattern para ISBN; xs:enumeration para género; xs:key para unicidad.

  • XML válido contra XSD.
  • Al menos 5 libros.
  • Entrega: biblioteca.xml, biblioteca.xsd.

XML · Eventos con namespaces

Enunciado: define <agenda> con <evento> (nombre, fecha, hora, lugar). Declara un namespace propio (por ejemplo, xmlns:ev="https://tu-centro/agenda") y usa atributos en prefijo para metadatos (p. ej., ev:categoria).

Pistas: decide qué va como atributo y qué como elemento; evita colisiones de nombres con prefijos.

Markdown · README técnico

Enunciado: documenta en Markdown uno de tus XML (contexto, estructura, decisiones de modelado, cómo validar). Incluye tablas, bloques de código y lista de tareas.

  • Archivo: README.md.
  • Debe incluir al menos una tabla y un bloque de código con sintaxis.

Markdown · Informe breve

Enunciado: redacta un informe comparando DTD vs XSD (fortalezas, limitaciones, casos de uso), con enlaces y citas.

Criterios de evaluación (resumen): corrección sintáctica (bien formado), calidad del modelo (elección de elementos/atributos), uso de validación (DTD/XSD), y calidad de la documentación (Markdown claro, completo y con ejemplos).
