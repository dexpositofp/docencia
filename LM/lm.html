Lenguaje de Marcas

Unidad didáctica — 1º DAW · Estructura, semántica y validación

¿Qué es un Lenguaje de Marcas?

Un lenguaje de marcas es un sistema que utiliza etiquetas o símbolos especiales (las “marcas”) para añadir significado y estructura a un contenido. De esta forma, un mismo texto puede enriquecerse con información adicional que lo hace comprensible tanto para las personas como para los programas informáticos.

Los lenguajes de marcas permiten definir qué partes de un documento corresponden a títulos, párrafos, imágenes, autores, fechas o cualquier otra entidad significativa. No son lenguajes de programación: no calculan, sino que describen. La clave está en que con marcado, un contenido plano se convierte en estructurado, semántico y procesable.

Idea clave: El marcado define qué es cada dato, mientras que otras tecnologías (CSS, JS) decidirán cómo se ve o cómo se comporta.

Tipos y usos habituales

  • HTML: lenguaje de marcas estándar de la web. Indica estructura, títulos, párrafos, listas, imágenes o formularios.
  • XML: lenguaje flexible para describir datos e intercambiarlos entre sistemas de manera normalizada.
  • Markdown: lenguaje ligero que busca legibilidad y rapidez al escribir documentación técnica o notas.

Comparativa rápida

LenguajeObjetivo principalVentajasLimitaciones
HTML Describir estructura y contenido de la web Estándar universal; semántica rica; compatible con todos los navegadores No se usa para datos arbitrarios; necesita CSS y JS para presentación e interacción
XML Modelar e intercambiar datos Extensible, validable, autodescriptivo Sintaxis extensa; requiere definir reglas adicionales (DTD/XSD)
Markdown Formatear texto simple y legible Curva de aprendizaje muy corta; ideal para documentación No es tan potente como HTML; distintas variantes según la herramienta

XML

XML (eXtensible Markup Language) es un lenguaje de marcas diseñado para describir, almacenar e intercambiar datos. A diferencia de HTML, no tiene un conjunto fijo de etiquetas: cada comunidad o aplicación define las suyas propias en función de sus necesidades (clientes, pedidos, libros, recetas, etc.). La gran ventaja de XML es su extensibilidad: se adapta a cualquier dominio.

Un documento XML no indica cómo se debe mostrar la información, sino qué significa cada parte. De esta manera, dos sistemas distintos pueden intercambiar datos de forma consistente y procesarlos automáticamente.

Normas básicas de XML

  • Todas las etiquetas deben abrirse y cerrarse correctamente: <nombre>Ana</nombre>.
  • Los elementos deben estar bien anidados: 
    <persona><nombre>Ana</nombre></persona> ✅
    <persona><nombre>Ana</persona></nombre> ❌
  • Debe existir un único elemento raíz que contenga todo el documento.
  • Los atributos deben escribirse entre comillas: <precio moneda="EUR">29.99</precio>.
  • XML distingue mayúsculas y minúsculas: <Autor> y <autor> son elementos distintos.
  • Los caracteres especiales (<, >, &) deben escaparse o escribirse dentro de CDATA.

Ejemplo correcto de XML

<?xml version="1.0" encoding="UTF-8"?>
<catalogo>
  <producto id="P001">
    <nombre>Teclado mecánico</nombre>
    <precio moneda="EUR">89.90</precio>
  </producto>
  <producto id="P002">
    <nombre>Ratón inalámbrico</nombre>
    <precio moneda="EUR">29.90</precio>
  </producto>
</catalogo>

Errores más comunes

Estos son algunos errores frecuentes que provocan que un XML no sea válido o ni siquiera “bien formado”:

  • Más de un elemento raíz: 
    <persona>...</persona>
<producto>...</producto> ❌
  • Etiqueta sin cerrar: 
    <autor>Juan Pérez ❌
  • Mal anidado: 
    <persona><nombre>Ana</persona></nombre> ❌
  • Atributos sin comillas: 
    <precio moneda=EUR>29.99</precio> ❌
  • Uso incorrecto de caracteres especiales: 
    <descripcion>5 < 10</descripcion> ❌

    Debe escribirse como: 5 &lt; 10

Consejo: utiliza siempre un editor que tenga resaltado de sintaxis o validador integrado (por ejemplo, VS Code con extensión XML Tools). Así detectarás rápidamente errores de anidado o etiquetas sin cerrar.

Bien formado vs válido

Un documento bien formado cumple todas las reglas sintácticas descritas arriba. Un documento válido, además, sigue un conjunto de reglas definidas en un DTD o un esquema XSD. En esta unidad nos centraremos en aprender a escribir XML bien formado y a validarlo contra un esquema sencillo.

Validación de XML

Validar un XML significa comprobar que su estructura y contenido cumplen un conjunto de reglas. Hay dos vías clásicas: DTD (sencillo y sintético) y XSD (potente, tipado y con namespaces). Un documento puede estar bien formado y, aun así, no ser válido si no respeta el DTD/XSD asociado.

1) DTD (Document Type Definition)

Un DTD define qué elementos existen, cómo se estructuran (orden y repetición) y qué atributos puede tener cada elemento. No hay tipos numéricos/fecha: todo es texto; por eso es ligero, pero menos expresivo que XSD.

1.1. DTD interno (en el propio XML)

<!DOCTYPE biblioteca [
  <!ELEMENT biblioteca (libro+)>             <!-- 1..n libros -->
  <!ELEMENT libro (titulo, autor, anio)>      <!-- orden estricto -->
  <!ATTLIST libro isbn ID #REQUIRED>          <!-- atributo único en el doc -->

  <!ELEMENT titulo (#PCDATA)>                 <!-- texto -->
  <!ELEMENT autor  (#PCDATA)>
  <!ELEMENT anio   (#PCDATA)>
]>
<biblioteca>
  <libro isbn="ISBN-001">
    <titulo>XML práctico</titulo>
    <autor>A. López</autor>
    <anio>2024</anio>
  </libro>
</biblioteca>

1.2. DTD externo (archivo .dtd)

Coloca las reglas en biblioteca.dtd y referencia desde el XML:

<!DOCTYPE biblioteca SYSTEM "biblioteca.dtd">
<biblioteca>...</biblioteca>

1.3. Modelos de contenido y cardinalidades

  • (a, b) → secuencia en ese orden.
  • (a | b) → o uno o el otro.
  • a? → 0..1; a* → 0..n; a+ → 1..n.
  • (#PCDATA) → texto; (#PCDATA | b | i)* → contenido mixto (texto + marcas).

1.4. Atributos en DTD

<!ATTLIST producto
  id     ID        #REQUIRED
  stock  CDATA     #IMPLIED
  estado (nuevo|oferta|descatalogado) "nuevo"
>
  • ID debe ser único en todo el documento; IDREF/IDREFS referencian un ID existente.
  • Enumeraciones fijan el valor a un conjunto cerrado.
  • #REQUIRED, #IMPLIED (opcional) o valor por defecto.

1.5. Contenido mixto (ejemplo clásico)

<!ELEMENT p (#PCDATA | b | i | enlace)*>

Errores típicos en DTD: ordenar mal los hijos (la secuencia importa), olvidar cerrar etiquetas en el XML, repetir ID, y confundir (a|b) con (a,b).

2) XML Schema (XSD)

XSD valida estructura y, además, tipa valores (números, fechas, booleanos), define restricciones como longitudes, rangos y patrones (regex), admite namespaces y claves/relaciones (key/keyref).

2.1. Estructura mínima de un XSD

<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema"
           elementFormDefault="qualified">

  <!-- Elemento raíz -->
  <xs:element name="biblioteca">
    <xs:complexType>
      <xs:sequence>
        <xs:element name="libro" type="LibroType" maxOccurs="unbounded"/>
      </xs:sequence>
    </xs:complexType>
  </xs:element>

  <!-- Tipo complejo con hijos y atributo tipado -->
  <xs:complexType name="LibroType">
    <xs:sequence>
      <xs:element name="titulo" type="xs:string"/>
      <xs:element name="autor"  type="xs:string"/>
      <xs:element name="anio"   type="xs:gYear"/>
    </xs:sequence>
    <xs:attribute name="isbn" type="isbnType" use="required"/>
  </xs:complexType>

  <!-- Tipo simple restringido mediante patrón -->
  <xs:simpleType name="isbnType">
    <xs:restriction base="xs:string">
      <xs:pattern value="ISBN-\d{3}"/> <!-- p. ej., ISBN-123 -->
    </xs:restriction>
  </xs:simpleType>

</xs:schema>

2.2. Reglas de ocurrencia

  • minOccurs / maxOccurs (p. ej., maxOccurs="unbounded").
  • Composición: xs:sequence (orden), xs:choice (alternativa), xs:all (sin orden, 0..1 cada hijo).

2.3. Restringir valores (facets)

<xs:simpleType name="precioType">
  <xs:restriction base="xs:decimal">
    <xs:minInclusive value="0.00"/>
    <xs:fractionDigits value="2"/>
  </xs:restriction>
</xs:simpleType>

<xs:simpleType name="generoType">
  <xs:restriction base="xs:string">
    <xs:enumeration value="Tecnico"/>
    <xs:enumeration value="Novela"/>
    <xs:enumeration value="Ensayo"/>
  </xs:restriction>
</xs:simpleType>

2.4. Claves y referencias (unicidad e integridad)

<xs:element name="pedido">
  <xs:complexType>
    <xs:sequence>
      <xs:element name="linea" maxOccurs="unbounded">
        <xs:complexType>
          <xs:sequence>
            <xs:element name="productoId" type="xs:ID"/>
            <xs:element name="cantidad"   type="xs:positiveInteger"/>
          </xs:sequence>
        </xs:complexType>
      </xs:element>
    </xs:sequence>
  </xs:complexType>

  <!-- Unicidad: productoId único entre las líneas -->
  <xs:unique name="u_producto">
    <xs:selector xpath="linea"/>
    <xs:field xpath="productoId"/>
  </xs:unique>
</xs:element>

2.5. Namespaces y xsi:schemaLocation

En XML con namespaces, asocia cada URI a un fichero XSD con xsi:schemaLocation (o xsi:noNamespaceSchemaLocation si no hay namespace): 

<catalogo xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
          xsi:noNamespaceSchemaLocation="catalogo.xsd">
  ...
</catalogo>

Errores típicos en XSD: olvidar elementFormDefault="qualified" al usar namespaces, confundir sequence con choice, patrones muy permisivos, y usar string donde conviene un tipo específico (decimal, date, gYear…).

3) Guía rápida: de cero a validado

  1. Define el modelo en papel: elementos, orden, opcionales, repetición, atributos.
  2. Escribe un XML de ejemplo (bien formado).
  3. Crea el DTD/XSD:
    • DTD → modela estructura y atributos.
    • XSD → además, tipa y restringe valores con facets.
  4. Referencia el DTD/XSD desde el XML.
  5. Valida y corrige hasta que no haya errores.

4) Mini-ejemplo completo (XSD + XML)

catalogo.xsd (tipado básico y patrón) 
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema"
           elementFormDefault="qualified">

  <xs:element name="catalogo">
    <xs:complexType>
      <xs:sequence>
        <xs:element name="producto" type="ProductoType" maxOccurs="unbounded"/>
      </xs:sequence>
    </xs:complexType>
  </xs:element>

  <xs:complexType name="ProductoType">
    <xs:sequence>
      <xs:element name="nombre" type="xs:string"/>
      <xs:element name="precio" type="precioType"/>
    </xs:sequence>
    <xs:attribute name="id" type="idType" use="required"/>
  </xs:complexType>

  <xs:simpleType name="precioType">
    <xs:restriction base="xs:decimal">
      <xs:minInclusive value="0.00"/>
      <xs:fractionDigits value="2"/>
    </xs:restriction>
  </xs:simpleType>

  <xs:simpleType name="idType">
    <xs:restriction base="xs:string">
      <xs:pattern value="P-\d{3}"/>  <!-- P-001, P-123 -->
    </xs:restriction>
  </xs:simpleType>
</xs:schema>
catalogo.xml (válido contra el XSD) 
<catalogo xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
          xsi:noNamespaceSchemaLocation="catalogo.xsd">
  <producto id="P-001">
    <nombre>Teclado mecánico</nombre>
    <precio>89.90</precio>
  </producto>
  <producto id="P-002">
    <nombre>Ratón inalámbrico</nombre>
    <precio>29.90</precio>
  </producto>
</catalogo>

5) Checklist de depuración

  • Raíz única y etiquetas bien anidadas/cerradas.
  • ¿El orden de los hijos coincide con sequence (XSD) o el modelo (DTD)?
  • ¿Cardinalidades correctas (minOccurs/maxOccurs, * + ? en DTD)?
  • ¿Atributos requeridos presentes? ¿Enumeraciones respetadas?
  • ¿Tipos correctos (número/fecha) y patrones (pattern) bien definidos?
  • ¿Referencia al esquema correcta (schemaLocation / DOCTYPE)?

Recurso oficial: para consultar todas las construcciones de XSD (tipos, facets, claves…), usa la documentación de XML Schema 1.1 del W3C. A nivel de curso, con lo anterior podrás modelar y validar la mayoría de casos.

Markdown

Markdown es un lenguaje de marcado ligero creado en 2004 por John Gruber con la idea de que los documentos sean fáciles de leer en texto plano y, al mismo tiempo, se conviertan sin esfuerzo a HTML. Su éxito radica en su sencillez: en pocos minutos puedes aprender a dar formato a tus textos.

Hoy en día, Markdown es omnipresente en el trabajo diario de cualquier desarrollador o estudiante de informática:

  • Documentación de proyectos: todos los repositorios en GitHub, GitLab o Bitbucket usan README.md como presentación del proyecto.
  • Wikis y apuntes: plataformas como Notion, Obsidian o HackMD usan sintaxis Markdown para notas técnicas.
  • Blogs y publicaciones: motores de blog como Jekyll, Hugo o Hexo renderizan ficheros Markdown a HTML.
  • Comunicación en equipo: Slack, Discord o gestores de incidencias como Jira/Redmine aceptan sintaxis Markdown para mensajes enriquecidos.
  • Exámenes y prácticas: muchos profesores piden entregas de documentación en Markdown por su portabilidad a PDF, HTML o Word (con herramientas como Pandoc).

Idea clave: Markdown no pretende sustituir a HTML, sino simplificar la escritura de documentos técnicos o de texto formateado sin tener que preocuparse por etiquetas complejas.

Sintaxis básica

Los elementos más comunes de Markdown son muy fáciles de recordar:

# Encabezado nivel 1
## Encabezado nivel 2
### Encabezado nivel 3

Este es un **texto en negrita**, este en *cursiva* y este en `código inline`.

- Lista no ordenada
- Otro elemento
1. Lista ordenada
2. Segundo elemento

[Enlace a un recurso](https://developer.mozilla.org/es/)

Extensiones útiles (GitHub Flavored Markdown)

  • Tablas: 
    | Nombre   | Edad | Ciudad   |
|----------|------|----------|
| Ana      |  23  | Sevilla  |
| Roberto  |  30  | Madrid   |
  • Listas de tareas (ideales para seguimiento de proyectos): 
    - [x] Configurar repositorio
- [x] Escribir README inicial
- [ ] Añadir pruebas unitarias
  • Bloques de código con resaltado: 
    ```javascript
function hola() {
  console.log("¡Hola, mundo!");
}
```
  • Citas y avisos: 
    > Esto es una cita o nota destacada.
> Puede ocupar varias líneas.

Buenas prácticas

  • Empieza siempre tus proyectos con un README.md claro: qué hace, cómo instalarlo y cómo usarlo.
  • Divide el contenido en secciones con encabezados jerárquicos #, ##, ###.
  • Usa tablas y listas de tareas para mostrar estado de desarrollo o requisitos.
  • Incluye ejemplos de uso en bloques de código con resaltado de sintaxis.
  • Añade enlaces a documentación oficial o recursos externos.

Consejo: en tu día a día como programador, documenta todo en Markdown: desde tus apuntes hasta las guías de instalación de un proyecto. Ganarás en claridad y portabilidad.

Ejemplo de README.md típico

# Proyecto Lista de la Compra

Aplicación web sencilla que permite crear, editar y eliminar elementos de una lista de la compra.

## Instalación
1. Clona el repositorio
2. Abre `index.html` en tu navegador

## Uso
- Añade un ítem en el campo de texto
- Haz clic en **Añadir** para guardarlo
- Puedes eliminar o marcar como comprado cada elemento

## Tecnologías
- HTML5
- CSS3
- JavaScript (ES6)

## Autor
María López · 1º DAW

Ejercicios propuestos

  1. Crea un documento README.md para un proyecto ficticio explicando el objetivo, instalación, uso y tecnologías empleadas.
  2. Redacta una tabla en Markdown con tus tres lenguajes de programación favoritos y su nivel (básico, intermedio, avanzado).
  3. Haz una lista de tareas en Markdown para planificar tu semana (al menos 5 elementos, algunos marcados como completados).
  4. Escribe una cita de un autor que te inspire usando la sintaxis de cita > en Markdown.
  5. Escribe un bloque de código con resaltado de sintaxis mostrando un ejemplo sencillo en el lenguaje que prefieras.

Referencia: la especificación moderna de Markdown es CommonMark. La variante más usada hoy en día es GitHub Flavored Markdown (GFM), documentada en github.github.com/gfm.

Ejercicios propuestos

  1. Crea un documento HTML con un título y una lista de tres elementos. Usa etiquetas semánticas.
  2. Diseña un XML para representar una agenda de eventos con charlas (título, autor, hora). Comprueba que sea bien formado.
  3. Define un DTD para validar el XML de la agenda, obligando a que cada evento tenga título y hora.
  4. Crea un XML de biblioteca y un XSD que restrinja el año a cuatro dígitos y el ISBN a un formato concreto.
  5. Redacta un README en Markdown que documente tu XML: incluye descripción, tabla de elementos y un bloque de código.

Criterios de evaluación: corrección sintáctica, validez según DTD/XSD, organización clara de etiquetas, y documentación en Markdown clara y completa.
