HTML

Structure and meaning of the content of a web page

Introduction to HTML

HTML (HyperText Markup Language) is the standard for structuring documents on the web. Together with CSS (presentation) and JavaScript (interactivity), it forms the technological foundation of the front-end.

What is HTML?

It is a markup language: it does not program, but labels content to describe its function (paragraphs, lists, images, tables, forms…). Browsers interpret these tags to build the document tree (DOM) and render it.

Meaning of the acronym

  • Hyper: hyperlinks that connect resources to each other.
  • Text: the content that is presented (text, image, multimedia).
  • Markup: semantic tags that give structure and meaning.
  • Language: formal set of elements and attributes with defined syntax.

History (very brief) and HTML5

Since 1991, HTML has evolved into HTML5, which includes semantic tags (header, nav, main, section, article…), enhanced forms, and modern browser APIs (multimedia, canvas, etc.).

Minimal example

<!DOCTYPE html>
    <html lang="en">
    <head>...</head>
    <body>
      <h1>Hello, world!</h1>
      <p>First HTML document.</p>
    </body>
    </html>

Separation of concerns: HTML (structure and meaning), CSS (presentation), JS (behavior). This separation improves accessibility, maintenance, and testing.

Syntax and Basic Tags

An HTML element consists of an opening tag, content, and (usually) a closing tag. Attributes add metadata or behavior. Always use semantic tags when possible, and reserve <div>/<span> for when there's no better alternative.

Text, Headings, and Links

  • <h1..h6>: heading hierarchy. Usually one <h1> per page or main content block.
  • <p>: paragraphs; <strong>/<em> for emphasis (semantic, not visual).
  • <a href>: links with meaningful text (avoid “click here”). If opening in a new tab, add rel="noopener noreferrer".
<h1>Travel Guide</h1>
    <p>Check the <a href="#tips">pre-travel tips</a> before booking.</p>
    <p>Deals are <strong>available</strong> until the end of the month.</p>

Lists: Unordered, Ordered, and Definition

Choose the list type based on meaning, not appearance:

  • Unordered (<ul>): a group with no inherent order.
  • Ordered (<ol>): steps or a hierarchy with order.
  • Definition (<dl>): terms and their descriptions.
<h2>Travel Checklist</h2>
    <ul>
      <li>Passport</li>
      <li>Health card</li>
      <li>Plug adapter</li>
    </ul>
    
    <h2>Steps to Book</h2>
    <ol>
      <li>Choose destination</li>
      <li>Compare flights</li>
      <li>Confirm accommodation</li>
    </ol>
    
    <h2>Key Terms</h2>
    <dl>
      <dt>Check-in</dt><dd>Check-in at the accommodation.</dd>
      <dt>Stopover</dt><dd>An intermediate stop between flights.</dd>
    </dl>

Tip: nesting lists is fine when the hierarchy makes sense, but avoid deep levels (they lose clarity).

Images (alt, dimensions, responsive)

  • <img src alt>: alt describes the image; if decorative, use alt="".
  • width/height help avoid layout shifts.
  • Responsive: use srcset/sizes or <picture> for alternative formats.
<figure>
      <img
        src="img/beach-800.jpg"
        srcset="img/beach-400.jpg 400w, img/beach-800.jpg 800w, img/beach-1200.jpg 1200w"
        sizes="(max-width: 600px) 90vw, 800px"
        width="800" height="533"
        alt="Beach with clear sky and umbrellas">
      <figcaption>View of the beach at sunset.</figcaption>
    </figure>

Correct alt: what the image is and why it’s there. Avoid “image of…”. Describe relevant info for the context.

Paths and URLs in src/href

  • Relative (e.g., ../img/logo.svg).
  • Absolute (e.g., https://example.com).
  • Fragment (e.g., #contact).

Neutral Containers (use wisely)

  • <div>: block-level with no semantics (for grouping).
  • <span>: inline with no semantics (for small fragments inside text).

Golden rule: if there’s a specific semantic tag for your case (e.g., <header> instead of <div class="header">), use it.

Semantic Sectioning Tags (HTML5)

These tags give structure and meaning to the document. They improve accessibility, SEO, and maintainability. They do not change appearance by themselves (that's what CSS does), but they create a clear semantic “spine.”

<main>

The unique main content of the page. It should appear only once and should not contain other landmark regions such as the site’s header or footer.

<header> and <footer>

  • Page-level: global site header and footer.
  • Section or article-level: you can have local headers/footers within section/article for titles, metadata, local navigation, etc.

<nav>

Navigation block with primary or secondary links. There can be more than one (global nav, breadcrumb nav, section nav).

<section> vs <article>

  • <article>: a self-contained, reusable unit of content (post, product card, news). It makes sense out of context.
  • <section>: groups related content with its own heading, but depends on the page context.

<aside>

Content tangential to the main content (sidebars, promos, glossaries). In screen readers, it's announced as complementary.

<figure> and <figcaption>

For images, charts, code snippets, or tables that need a caption or explanation.

<time datetime>

Machine-readable date/time (useful for SEO, microformats, and sorting).

Semantic Page Pattern

<header>
      <h1>Site Name</h1>
      <nav>
        <ul><li><a href="/">Home</a></li>...</ul>
      </nav>
    </header>
    
    <main>
      <article>
        <header>
          <h2>Article Title</h2>
          <p><time datetime="2025-10-18">Oct 18, 2025</time> — By Ana</p>
        </header>
    
        <section>
          <h3>Introduction</h3>
          <p>...</p>
        </section>
    
        <section>
          <h3>Development</h3>
          <figure>
            <img src="img/diagram.svg" alt="System workflow diagram">
            <figcaption>Workflow diagram.</figcaption>
          </figure>
        </section>
    
        <footer>
          <p>Tags: <a href="/tags/html">HTML</a>, <a href="/tags/accessibility">Accessibility</a></p>
        </footer>
      </article>
    
      <aside>
        <h2>Related</h2>
        <ul>
          <li><a href="/article-1">Good ALT Practices</a></li>
        </ul>
      </aside>
    </main>
    
    <footer>
      <small>© 2025 — School Name</small>
    </footer>

Headings and outline: use h1..h6 levels hierarchically. A typical pattern is: 1× h1 for the page title, then h2 for sections, h3 for subsections, and so on.

Useful Native Widgets

  • <details> + <summary> for accessible accordions without JS.
  • <progress> / <meter> to represent progress/values.
  • <dialog> (with some JS) for accessible modals.
<details>
      <summary>View minimum requirements</summary>
      <ul><li>Modern browser</li><li>Stable connection</li></ul>
    </details>

Avoid using <div> with class names like header/footer/section when real semantic tags exist. It improves landmark navigation for screen readers and clarifies the DOM.

Web Design Principles

They guide the user experience: usability (clear flow), consistency (patterns), readability (typography, contrast), accessibility (content for everyone), and performance (reasonable load times).

  • Navigation: predictable placement, visual feedback on state, accessible menus.
  • Typography: relative sizes, generous line heights, reasonably short paragraphs.
  • Color: sufficient contrast (WCAG); don’t rely on color alone to convey meaning.

Tip: design the information flow first; the styling should enhance it, not hide it.

Usability and Accessibility

Usability means effectiveness + efficiency + satisfaction. Accessibility means content is usable by people with different abilities (keyboard users, screen readers, low vision, etc.).

  • Use landmarks (header/nav/main/footer), hierarchical headings, and proper alt text.
  • In forms: <label for> + id, required, and associated messages (aria-describedby).
  • Focus management: logical sequence, visible focus, and ARIA roles only when native semantics are missing.
<a class="skip-link" href="#main">Skip to content</a>

Forms

They collect user data. A good form combines semantics, accessibility, client-side validation (HTML5), and server-side validation.

Basic Structure

The <form> tag wraps the controls. Always use name to send data and label for accessibility.

<form action="/submit" method="post" novalidate>
      <div>
        <label for="nombre">Name</label>
        <input id="nombre" name="nombre" type="text" required autocomplete="name">
      </div>
      <div>
        <label for="email">Email</label>
        <input id="email" name="email" type="email" required autocomplete="email">
      </div>
      <button type="submit">Send</button>
    </form>

method: get for queries or filters (parameters in URL); post for submissions. action: path for processing.

Useful Attributes and States

  • required, min, max, minlength, maxlength, step, pattern
  • CSS pseudo-classes: :required, :optional, :valid, :invalid, :in-range, :out-of-range
  • Help text using aria-describedby managed by id.
<label for="tel">Phone (9 digits)</label>
    <input id="tel" name="tel" type="tel" pattern="[0-9]{9}" required
           aria-describedby="help-tel">
    <small id="help-tel" class="muted">Numbers only, no spaces.</small>

Grouping and Assistance

To structure and improve accessibility:

  • <fieldset> groups related controls + <legend> describes the group.
  • <optgroup> organizes options within <select>.
  • <datalist> suggests values to an <input list>.
<fieldset>
      <legend>Personal Data</legend>
      <label for="name">Name</label>
      <input id="name" name="name" type="text">
      <label for="city">City</label>
      <input list="cities" id="city" name="city">
      <datalist id="cities">
        <option value="Madrid">
        <option value="Valencia">
      </datalist>
    </fieldset>

Browser Validation

HTML5 implements native validation: formats (email, url), patterns (regex), required fields, ranges, etc. This is complementary to server-side validation.

<input type="email" name="email" required>
    <input type="text" name="phone" pattern="[0-9]{9}" required>

Important: never rely solely on client-side validation; always validate and sanitize on the server.

Form Practice Exercises

  • Registration form with name, surname, email, password, and confirmation (with required and format help).
  • Appointment booking: date, time, and service list (select/radio).
  • Image upload: accept only .jpg and .png (accept attribute).
  • Survey with checkbox and radio plus a comment field.
  • Contact form with email and phone validation using pattern.

Lists in HTML

Lists allow you to group related content and present it in a structured, semantic way. HTML provides three main types of lists, each suited for different content types:

Unordered Lists (<ul>)

Used when the order of items does not matter. Each item is wrapped in an <li> tag.

<ul>
      <li>Apples</li>
      <li>Oranges</li>
      <li>Bananas</li>
    </ul>

Ordered Lists (<ol>)

Use these when the order of the items is important (e.g., steps, rankings).

<ol>
      <li>Register an account</li>
      <li>Log in</li>
      <li>Start learning</li>
    </ol>

Description Lists (<dl>)

Best for terms and definitions or any label-description pair. Uses <dt> (term) and <dd> (description).

<dl>
      <dt>HTML</dt>
      <dd>The standard markup language for web pages.</dd>
    
      <dt>CSS</dt>
      <dd>A language to describe the style of an HTML document.</dd>
    </dl>

Tip: Choose your list type based on the meaning of the content, not on how it looks. Use CSS to control presentation, not semantics.

Tables and Multimedia

Accessible Tables

Use them for tabular data, not for layout. Include <caption>, headers, and appropriate scope.

<table>
      <caption>Quarterly Results</caption>
      <thead>
        <tr><th scope="col">Month</th><th scope="col">Sales</th></tr>
      </thead>
      <tbody>
        <tr><th scope="row">January</th><td>120k</td></tr>
      </tbody>
    </table>

Multimedia in HTML5

  • <audio> and <video> with controls, preload, and caption tracks (<track kind="captions">).
  • <canvas> for JS-rendered graphics; <svg> for semantic vector graphics.
<video width="480" controls preload="metadata">
      <source src="movie.mp4" type="video/mp4">
      <track src="sub.vtt" kind="captions" srclang="es" label="Spanish">
      Your browser does not support video.
    </video>
<canvas id="chart" width="240" height="120"></canvas>
    <script>
      const ctx = document.getElementById('chart').getContext('2d');
      ctx.fillRect(10, 10, 80, 40);
    </script>

HTML Document Validation

Validation helps catch markup errors and improve compatibility. Make it part of your workflow (validators, linters, testing across browsers/screen readers).

Quick checklist: correct lang and charset, proper semantics, good contrast, images with alt, labeled forms, and no errors in the W3C validator.

XHTML Restrictions

XHTML is HTML rewritten as XML. It requires well-formed documents: lowercase tags, required closing tags, quoted attributes, and a single root element. While its strictness promoted interoperability and XML tooling, HTML5 now dominates. Still, "well-formed" rules are useful as formal discipline.

Basic HTML Exercises

  1. Personal Introduction

    Include an <h1> with your name, a paragraph with your interests, and a list of favorites.

  2. Basic Structure

    Create a minimal document with <!DOCTYPE>, <html>, <head>, and <body>, using lang="es", charset, and viewport.

  3. Link and Image

    Add an <a> with meaningful text and a centered <img> with descriptive alt.

  4. Basic Form

    Name, email, comments, and a submit button, using label and required.

  5. Simple Table

    3×3 table with headers in the first row and a <caption>.
